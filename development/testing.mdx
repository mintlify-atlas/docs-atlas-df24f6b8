## Overview

Salud uses a multi-layered testing approach to ensure reliability and security across the frontend, backend, and smart contracts.

## Testing Philosophy

### Quality Standards

- **Smart Contracts**: Comprehensive test coverage (security-critical)
- **Backend**: API endpoint validation and error handling
- **Frontend**: Build verification and linting
- **Integration**: Manual end-to-end testing

### Test Pyramid

```
        /\        E2E Tests (Manual)
       /  \       - User flows
      /    \      - Cross-browser
     /      \     
    /--------\    Integration Tests (Manual/Automated)
   /          \   - API endpoints
  /            \  - Contract interactions
 /              \ 
/----------------\ Unit Tests (Automated)
                   - Contract transitions
                   - Utility functions
                   - Component rendering
```

<CodeGroup>

The contract test suite (`tests/test_salud_health_records.leo`) covers:

#### Record Creation Tests

<AccordionGroup>
  <Accordion title="test_create_record">
    Verifies basic record creation with valid inputs.
    
    **Tests:**
    - Record is created successfully
    - Owner is set to caller
    - Data fields are stored correctly
    - Record type is valid
  </Accordion>

  <Accordion title="test_record_types">
    Validates all 10 record type categories.
    
    **Record Types Tested:**
    1. General Health
    2. Laboratory Results
    3. Prescriptions
    4. Imaging/Radiology
    5. Vaccination Records
    6. Surgical Records
    7. Mental Health
    8. Dental Records
    9. Vision/Ophthalmology
    10. Other/Miscellaneous
  </Accordion>

  <Accordion title="test_invalid_record_type">
    Ensures invalid record types (0, 11+) are rejected.
    
    **Expects:** Assertion failure for out-of-range types
  </Accordion>
</AccordionGroup>

#### Record ID Tests

<AccordionGroup>
  <Accordion title="test_record_id_uniqueness">
    Verifies that different inputs produce unique record IDs.
    
    **Tests:**
    - Different data → different IDs
    - Different nonces → different IDs
    - Different patients → different IDs
  </Accordion>

  <Accordion title="test_record_id_determinism">
    Confirms same inputs always produce same ID.
    
    **Tests:**
    - Same patient + data + nonce = same ID
    - Useful for client-side ID computation
  </Accordion>
</AccordionGroup>

#### Access Grant Tests

<AccordionGroup>
  <Accordion title="test_grant_access">
    Tests basic access grant creation.
    
    **Verifies:**
    - Access token is generated
    - Grant is stored in mapping
    - Patient retains record ownership
  </Accordion>

  <Accordion title="test_access_token_generation">
    Validates access token uniqueness.
    
    **Tests:**
    - Different doctors → different tokens
    - Different nonces → different tokens
    - Same inputs → same token (deterministic)
  </Accordion>

  <Accordion title="test_duration_bounds">
    Checks duration clamping to valid ranges.
    
    **Tests:**
    - Minimum: 240 blocks (~1 hour)
    - Maximum: 40,320 blocks (~7 days)
    - Out-of-range values are clamped
  </Accordion>
</AccordionGroup>

#### Access Verification Tests

<AccordionGroup>
  <Accordion title="test_verify_valid_access">
    Tests successful verification with valid token.
    
    **Flow:**
    1. Create record
    2. Grant access to doctor
    3. Verify access succeeds
  </Accordion>

  <Accordion title="test_verify_invalid_token">
    Ensures fake tokens are rejected.
    
    **Tests:**
    - Random token → verification fails
    - Wrong token for record → fails
  </Accordion>

  <Accordion title="test_verify_wrong_doctor">
    Prevents access by unauthorized doctors.
    
    **Scenario:**
    - Grant to Doctor A
    - Doctor B tries to access
    - Verification fails
  </Accordion>

  <Accordion title="test_verify_wrong_record">
    Ensures tokens are record-specific.
    
    **Scenario:**
    - Token for Record A
    - Used on Record B
    - Verification fails
  </Accordion>
</AccordionGroup>

#### Revocation Tests

<AccordionGroup>
  <Accordion title="test_revoke_access">
    Tests immediate access revocation.
    
    **Flow:**
    1. Grant access
    2. Revoke access
    3. Verification fails
    4. Access info shows `is_revoked: true`
  </Accordion>

  <Accordion title="test_full_flow">
    End-to-end integration test.
    
    **Complete Flow:**
    1. Patient creates record
    2. Patient grants access to doctor
    3. Doctor verifies access (succeeds)
    4. Patient revokes access
    5. Doctor verification fails
  </Accordion>
</AccordionGroup>

#### Helper Function Tests

<AccordionGroup>
  <Accordion title="test_compute_record_id">

    **Purpose:** Allows frontend to predict record IDs
  </Accordion>

  <Accordion title="test_get_access_info">
    Tests public access info retrieval.
    
    **Returns:**
    - Patient address
    - Doctor address
    - Record ID
    - Expiration time
    - Revocation status
  </Accordion>
</AccordionGroup>

### Writing New Contract Tests

Example test structure:

```leo
// tests/test_salud_health_records.leo

@test
function test_your_feature() {
    // Setup: Create test data
    let patient: address = aleo1...;
    let doctor: address = aleo1...;
    
    // Execute: Call transition
    let record: MedicalRecord = create_record(
        1field, 2field, 3field, 4field,
        1u8,      // record_type
        100field, // data_hash
        42field,  // nonce
        false     // make_discoverable
    );
    
    // Assert: Verify results
    assert_eq(record.owner, patient);
    assert_eq(record.record_type, 1u8);
    assert(record.record_id != 0field);
}
```

**Best Practices:**
- Test one thing per test function
- Use descriptive function names
- Add comments explaining what's being tested
- Test both success and failure cases
- Verify security constraints

  <Step title="Start the Server">

    ```json
    {
    ```
    ```json
      "success": true,
      "address": "aleo1...",
      "sessionId": "uuid-here"
    ```

  <Step title="Test Record Creation">

## Integration Testing

### End-to-End User Flows

<Steps>
  <Step title="Patient Creates Record">
    **Test Flow:**
    1. Open app at `http://localhost:5173`
    2. Connect wallet (or generate new)
    3. Click "New Record" button
    4. Enter medical data
    5. Submit record
    6. Verify success message
    7. See record in list
    
    **Expected Result:** Record appears in dashboard
  </Step>

  <Step title="Patient Shares Record">
    **Test Flow:**
    1. Click "Share" on a record
    2. Enter doctor address (or use default duration)
    3. QR code displays
    4. Verify expiration countdown shows
    5. Screenshot QR code for doctor test
    
    **Expected Result:** Valid QR code generated
  </Step>

  <Step title="Doctor Scans and Views">
    **Test Flow:**
    1. Navigate to `/doctor` route
    2. Connect doctor wallet
    3. Click "Scan QR Code"
    4. Grant camera permission
    5. Scan patient's QR code
    6. View decrypted record
    7. Verify expiration timer
    
    **Expected Result:** Record displays with countdown
  </Step>

  <Step title="Access Expiration">
    **Test Flow:**
    1. Create record with 1-hour duration
    2. Share with doctor
    3. Wait for expiration (or modify block height in demo)
    4. Doctor attempts to view
    5. Verify "Access expired" error
    
    **Expected Result:** Access denied after expiration
  </Step>
</Steps>

## Testing Best Practices

### Before Committing

<Steps>
  <Step title="Run All Checks">
    ```bash
    # Frontend
    cd frontend
    npm run lint
    npm run build
    
    # Backend
    cd ../backend
    npm start  # Verify starts
    # Ctrl+C to stop
    
    # Contract (if changed)
    cd ../salud_health_records
    leo test
    ```
  </Step>

  <Step title="Manual Testing">
    - Test the feature you changed
    - Test related features
    - Check for console errors
    - Verify mobile responsive (if UI change)
  </Step>

  <Step title="Document Changes">
    - Update documentation if needed
    - Add comments for complex logic
    - Update API docs if endpoints changed
  </Step>
</Steps>

<Checklist>
- [ ] All automated tests pass

- [ ] Console has no errors
- [ ] Works on Chrome and Safari
- [ ] Mobile responsive (for UI changes)
